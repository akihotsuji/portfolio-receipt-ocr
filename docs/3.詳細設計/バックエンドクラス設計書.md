## 3. レイヤー規約

各機能モジュール内の4レイヤーの責務と依存ルール。

```
presentation → application → domain ← infrastructure
```


| レイヤー           | 責務                               | 依存先                         |
| -------------- | -------------------------------- | --------------------------- |
| presentation   | HTTP リクエストの受付・レスポンス返却。バリデーション    | application                 |
| application    | ユースケースの実行。トランザクション管理             | domain, 他モジュールの application |
| domain         | ビジネスルール。モデルと Repository インターフェース | なし（最内層）                     |
| infrastructure | 技術的実装（MyBatis, SQS, Tesseract 等） | domain（インターフェースを実装）         |


**重要な規約:**

- `domain` パッケージは他のレイヤーに依存しない
- `infrastructure` は `domain` のインターフェースを実装する（依存性逆転）
- `presentation` の DTO と `domain` のモデルは分離する
- レイヤーをまたぐ依存は上方向のみ（presentation → application → domain）

## 4. 横断的関心事

### 4.1 infrastructure/（横断的インフラ）

機能モジュールをまたいで使用される技術的実装。


| クラス                     | パッケージ              | 責務                                        |
| ----------------------- | ------------------ | ----------------------------------------- |
| JwtConfig               | security           | JWKS URL 等の JWT 設定値を保持                    |
| JwtValidator            | security           | JWT トークンの署名検証と Claims 抽出                  |
| JwtAuthenticationFilter | security           | リクエストごとに JWT を検証する Filter                 |
| SecurityConfig          | security           | Spring Security の設定（パス別の認可ルール）            |
| JwtValidationException  | security.exception | JWT 検証失敗時の例外                              |
| GlobalExceptionHandler  | web                | `@RestControllerAdvice` で例外をエラーレスポンスに変換   |
| ApiErrorResponse        | web                | エラーレスポンスの共通フォーマット（API 設計書 §2.4 準拠）        |

| SqsMessageSender        | messaging          | SQS へのメッセージ送信（OCR ジョブ・エクスポートジョブ共通）        |

| 項目     | 仕様                                                                          |
| ------ | --------------------------------------------------------------------------- |
| ランタイム  | Java 21（Corretto）                                                           |
| ハンドラー  | `com.portfolio.receiptocr.lambda.exportcsv.ExportCsvHandler::handleRequest` |
| トリガー   | SQS `receipt-export-queue`（イベントソースマッピング、バッチサイズ 1）                           |
| タイムアウト | 300 秒                                                                       |
| メモリ    | 512 MB                                                                      |
| 同時実行数  | `ReservedConcurrentExecutions: 3`                                           |
| DB 接続  | JDBC + HikariCP（Lambda 外で初期化しコネクションを再利用）                                    |
| S3 操作  | AWS SDK v2 S3Client                                                         |
| デプロイ   | AWS SAM（`template.yaml`）                                                    |


Spring Boot の MyBatis / Repository パターンは Lambda では使用しない。Lambda は軽量に保つため、素の JDBC を使い、依存ライブラリを最小限にする。これによりデプロイパッケージサイズとコールドスタート時間を抑える。

- `infrastructure.storage.StorageClient` — CSV ファイルのダウンロード時に使用
- `infrastructure.messaging.SqsMessageSender` — キュー投入

**Lambda 側の外部依存:**

- PostgreSQL — JDBC で直接接続（`export_jobs`, `export_job_receipts`, `receipts` テーブル）
- S3 — AWS SDK v2 で CSV ファイルを保存

## 6. モジュール間依存関係

```
uploadbatch ──→ ocrjob ──→ receipt ←── export (Spring Boot)
                                         ↑
                              Lambda (export-csv) ─── SQS 経由で起動
```


| 依存元                 | 依存先                       | 理由                        |
| ------------------- | ------------------------- | ------------------------- |
| uploadbatch         | ocrjob                    | アップロード時に OCR ジョブを作成・キュー投入 |
| ocrjob              | receipt                   | 確定時にレシートを作成               |
| export（Spring Boot） | SQS                       | エクスポートジョブ作成時にキュー投入        |
| Lambda（export-csv）  | DB（receipts, export_jobs） | CSV 生成に必要なデータを JDBC で直接取得 |


- すべて単方向依存で循環なし
- Spring Boot 内の依存は application 層の Service 経由で行う（Repository への直接アクセスはしない）
- `receipt` モジュールは他モジュールに依存しない（参照される側）
- Lambda は Spring Boot と同じ DB を共有するが、コード上の依存はない（独立プロジェクト）

## 7. MyBatis 永続化パターン

JPA とは異なる MyBatis 固有のパターンを採用する。Repository パターンによる抽象化は維持する。

### 7.1 JPA（HabitTracker）との比較


| 観点                | JPA（HabitTracker）                             | MyBatis（receipt-ocr）                 |
| ----------------- | --------------------------------------------- | ------------------------------------ |
| DB 操作の定義          | JpaRepository が自動生成                           | Mapper インターフェース + XML で SQL を記述      |
| DB ↔ オブジェクト変換     | Entity クラス + JPA アノテーション                      | XML の resultMap でマッピング               |
| infrastructure 配下 | Entity, JpaRepository, Mapper, RepositoryImpl | Mapper, RepositoryImpl               |
| 動的クエリ             | Specification / QueryDSL                      | XML の `<if>`, `<where>`, `<foreach>` |


### 7.2 レイヤー間の関係

```
domain/repository/
└── OcrJobRepository.java          # インターフェース（ドメインが定義）
        ↑ 実装
infrastructure/mybatis/
├── OcrJobMapper.java              # @Mapper（MyBatis が SQL を実行）
└── OcrJobRepositoryImpl.java      # Repository 実装（Mapper に委譲）

resources/mapper/
└── OcrJobMapper.xml               # SQL 定義 + resultMap
```

- **Repository（domain）**: ドメインが必要とする操作を定義。技術的な詳細を含まない
- **Mapper（infrastructure）**: `@Mapper` アノテーション付きインターフェース。MyBatis が XML と紐づけて SQL を実行
- **RepositoryImpl（infrastructure）**: Repository を実装し、Mapper に委譲。ドメインモデルへの変換が必要な場合はここで行う
- **XML（resources）**: SQL 定義と resultMap（DB 行 → ドメインモデルのマッピング）

### 7.3 ドメインモデルへの直接マッピング

MyBatis の resultMap により、DB の行をドメインモデルに直接マッピングする。JPA の Entity クラスに相当する中間オブジェクトは原則不要。

```xml
<!-- OcrJobMapper.xml -->
<resultMap id="ocrJobMap" type="com.portfolio.receiptocr.ocrjob.domain.model.OcrJob">
    <id property="id" column="id" />
    <result property="batchId" column="batch_id" />
    <result property="status" column="status" />
    <!-- ... -->
    <association property="ocrResult" javaType="OcrResult">
        <result property="date" column="ocr_date" />
        <result property="amount" column="ocr_amount" />
        <!-- ... -->
    </association>
</resultMap>
```

## 8. 設計判断の根拠

### 8.1 4モジュール分割の採用

**検討した代替案:** フローベースの分割（`uploadFlow` / `receiptFlow`）


| 観点       | 4モジュール（採用）                     | フローベース        |
| -------- | ------------------------------ | ------------- |
| 責務の明確さ   | 1モジュール = 1リソース = 1テーブル群で明確     | フローに複数リソースが混在 |
| API との対応 | パスのプレフィックスと1対1                 | 対応が曖昧         |
| 独立性      | receipt は他に依存しない。モジュール単体でテスト可能 | フロー間の境界が不明確   |


### 8.2 OCR ワーカーをモジュール内に配置

**検討した代替案:** 横断的な `worker/` パッケージ、または `infrastructure/messaging/` に集約


| 観点      | モジュール内（採用）                  | 横断パッケージ         |
| ------- | --------------------------- | --------------- |
| 凝集度     | OCR ワーカーは OCR ジョブの文脈に属する    | 異なる責務のワーカーが混在   |
| 依存の局所化  | ワーカーが使うクラスが同一モジュール内に収まる     | モジュール間の依存が増加    |
| 変更の影響範囲 | OCR 処理の変更が ocrjob モジュール内で完結 | 横断パッケージの変更が他に波及 |


エクスポートワーカーは Lambda に移行したため、この判断は OCR ワーカーのみに適用される。

### 8.3 OcrEngine インターフェースの導入

Tesseract に直接依存せず、`domain/service/OcrEngine` インターフェースを介して抽象化する。

- テスト時にモック実装に差し替え可能
- 将来的に OCR エンジンを変更する場合（Cloud Vision API 等）の影響を局所化
- 依存性逆転の原則に従い、ドメインが技術的実装に依存しない

### 8.4 Repository + Mapper の2層構造

MyBatis の `@Mapper` を直接 Service から使う案も検討したが、Repository インターフェースを挟む構成を採用した。


| 観点      | Repository + Mapper（採用）             | Mapper 直接利用                |
| ------- | ----------------------------------- | -------------------------- |
| テスタビリティ | Repository をモックすれば Service 単体テストが容易 | Mapper のモックは MyBatis 依存が混入 |
| 抽象化     | ドメイン層が MyBatis に依存しない               | `@Mapper` がドメインに漏れる        |
| コスト     | RepositoryImpl が増える（薄い委譲クラス）        | クラス数は少ない                   |


学習テーマとして「永続化の抽象化」を重視し、多少のボイラープレートを許容する判断。

### 8.5 エクスポートワーカーの Lambda 化

エクスポート処理を Spring Boot 内の `@SqsListener` から AWS Lambda に移行した。

**検討した代替案:** OCR ワーカーも含め全ワーカーを Lambda 化


| 観点       | エクスポートのみ Lambda 化（採用）         | 全ワーカー Lambda 化                 |
| -------- | ----------------------------- | ------------------------------ |
| 実現容易性    | CSV 生成はネイティブライブラリ不要で移行が容易     | Tesseract OCR のネイティブライブラリ統合が困難 |
| コールドスタート | 軽量な依存で数秒程度                    | Tesseract 込みのイメージは重く 10 秒以上    |
| 学習効率     | SQS → Lambda パターンを1つ実装するだけで十分 | 2つ実装しても学習効果の上乗せが限定的            |


### 8.6 Lambda プロジェクトの独立

Lambda は Spring Boot とコードを共有せず、独立したプロジェクトとした。

**検討した代替案:** 共通モジュール（Maven マルチモジュール）で DB モデルを共有


| 観点       | 独立プロジェクト（採用）                    | 共通モジュール                |
| -------- | ------------------------------- | ---------------------- |
| コールドスタート | 依存が少なくデプロイパッケージが小さい（数 MB）       | 共通モジュールの推移的依存で肥大化するリスク |
| 保守性      | Lambda 側の変更が Spring Boot に影響しない | 共通モジュールの変更が両方に波及       |
| コスト      | DB モデルの軽微な重複が発生                 | クラス共有でコード重複なし          |


DB テーブル構造が安定しているため、軽量モデルの重複を許容してデプロイ独立性を優先した。

### 8.7 素の Lambda vs Spring Cloud Function

**検討した代替案:** Spring Cloud Function で Lambda を実装


| 観点       | 素の Lambda（採用）         | Spring Cloud Function           |
| -------- | --------------------- | ------------------------------- |
| コールドスタート | 数秒（依存が少ない）            | 10〜20 秒（Spring コンテナの初期化）        |
| 実装の親和性   | AWS SDK の API を直接使用   | Spring の抽象化経由                   |
| デプロイサイズ  | 小さい（数 MB）             | 大きい（Spring Framework 全体を含む）     |
| 学習効果     | Lambda の素の API を理解できる | Spring に隠蔽されて Lambda の仕組みが見えにくい |


ポートフォリオのデモでコールドスタートの遅延が目立つのは不利なため、軽量な素の Lambda を採用した。
