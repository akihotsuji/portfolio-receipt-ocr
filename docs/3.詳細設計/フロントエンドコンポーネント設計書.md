# パッケージ・コンポーネント設計書（フロントエンド）

## 1. 概要

領収書 OCR アプリケーションのフロントエンド（React）のディレクトリ構成とコンポーネント一覧を定義する。

### 設計方針

| 項目 | 方針 |
|---|---|
| アーキテクチャ | Feature ベース構成（HabitTracker 踏襲） |
| Feature 分割 | 画面単位の3モジュール |
| 状態管理 | React Context（外部ライブラリ不使用） |
| ルーティング | React Router |
| スタイリング | Tailwind CSS |

### 技術スタック（要件定義書より）

| 項目 | 技術 |
|---|---|
| フレームワーク | React 19 + TypeScript |
| ビルドツール | Vite |
| CSS | Tailwind CSS |
| 状態管理 | React Context |
| テスト | Jest |
| HTTP クライアント | Axios |

## 3. ディレクトリ規約

| ディレクトリ | 責務 | 配置ルール |
|---|---|---|
| components/ | 機能に依存しない共通 UI コンポーネント | 2つ以上の Feature で使われるもの |
| features/{name}/ | 画面単位の機能モジュール | api, components, hooks, types で責務分離 |
| contexts/ | React Context によるグローバル状態 | アプリ全体で共有する状態 |
| hooks/ | Feature をまたぐ共通カスタムフック | 2つ以上の Feature で使われるもの |
| lib/ | 外部ライブラリの設定・ラッパー | Axios 等のインフラ設定 |
| pages/ | ルーティング対象のページコンポーネント | Feature のコンポーネントを組み合わせる薄いラッパー |
| types/ | アプリ全体で共有する型定義 | API レスポンスの共通型など |
| utils/ | 純粋な関数ユーティリティ | フォーマット、バリデーション等 |

**Feature モジュール内の規約:**

| サブディレクトリ | 責務 |
|---|---|
| api/ | Axios を使った API 呼び出し関数 |
| components/ | 画面を構成する UI コンポーネント |
| hooks/ | API 呼び出し・状態管理のカスタムフック |
| types/ | Feature 固有の型定義（レスポンス型、ステータス enum 等） |

### 5.2 ocrjob（画面B: バッチ履歴・処理状況・確認・編集）

バッチ履歴の一覧表示、バッチの処理進捗表示、OCR 結果の確認・編集・確定、失敗ジョブのリトライ。

画面B は2カラムレイアウトで構成される。左カラムにバッチ履歴サイドバー、右カラムに選択中バッチの詳細（既存のプログレス・ジョブ一覧・編集フォーム）を表示する。

#### コンポーネント一覧

| コンポーネント | 責務 | 画面デザイン対応 |
|---|---|---|
| BatchHistorySidebar | バッチ履歴一覧のサイドバー（固定幅 280px）。バッチ一覧を表示し、選択中のバッチをハイライト。スクロール可能 | BatchHistorySidebar |
| BatchHistoryItem | バッチ1件の行。ファイル数 + ステータスインジケーター（完了/処理中/失敗の件数）+ 作成日時を表示 | BatchHistoryItem1〜3 |
| ProgressSection | プログレスバー + ステータス別件数バッジの表示 | ProgressSection |
| StatusBadge | ステータスに応じた色付きバッジ（待機中/処理中/完了/確定済/失敗） | StatusBadges 内の badge1〜5 |
| JobList | ジョブ一覧のコンテナ（ヘッダー + 行リスト） | JobList |
| JobRow | 1ジョブ行。ステータスに応じて右側の表示を切り替え（ボタン/スピナー/エラー等） | Job-Confirmed〜Job-Failed |
| EditSection | 確認・編集エリアのコンテナ（タイトル + 2カラムレイアウト） | EditSection |
| ImagePreview | 署名付き URL で取得した領収書画像のプレビュー表示 | ImagePreview |
| ConfirmForm | OCR 読み取り結果の編集フォーム（5フィールド + 確定ボタン） | FormArea |
| OcrRawText | OCR 生テキストのトグル表示セクション（読み取り専用） | OcrRawText |

## 6. ページコンポーネント

Feature のコンポーネントを組み合わせてページを構成する薄いコンポーネント。

| ページ | パス | 責務 |
|---|---|---|
| UploadPage | `/upload` | DropZone + FileList + UploadButton を配置 |
| OcrJobPage | `/batch` および `/batch/:batchId` | 2カラムレイアウト。左: BatchHistorySidebar（常時表示）、右: batchId がある場合は ProgressSection + JobList + EditSection を表示、ない場合は「バッチを選択してください」の空状態を表示 |
| ReceiptListPage | `/receipts` | FilterBar + ReceiptTable + Pagination + ExportActionBar を配置 |

**ルーティング（App.tsx）:**

```
/upload          → UploadPage（画面A）
/batch           → OcrJobPage（画面B: バッチ未選択、サイドバーのみ表示）
/batch/:batchId  → OcrJobPage（画面B: 選択バッチの詳細表示）
/receipts        → ReceiptListPage（画面C）
/                → /upload にリダイレクト
```

`/batch` と `/batch/:batchId` は同じ `OcrJobPage` コンポーネントを使用する。batchId の有無で右カラムの表示を切り替える。

## 7. Context・グローバル状態

学習テーマ「React Context」を活かし、アプリ全体で共有する状態を Context で管理する。

| Context | 責務 | 提供する値 |
|---|---|---|
| NotificationContext | トースト通知の表示・管理 | `showSuccess()`, `showError()`, 通知リスト |

- 通知は画面上部にバー形式で表示し、数秒で自動消去
- 各 Feature のフック（useUpload, useConfirmJob 等）が成功/失敗時に呼び出す
- batchId は URL パラメータで管理するため Context は不要

## 10. 設計判断の根拠

### 10.1 画面単位の3モジュール分割

**検討した代替案:** バックエンドと同じ4モジュール（upload, ocrjob, receipt, export）

| 観点 | 3モジュール（採用） | 4モジュール |
|---|---|---|
| 画面との対応 | 1モジュール = 1画面で直感的 | export が画面C に属するのに独立モジュール |
| コンポーネントの凝集度 | ExportActionBar が receipt 一覧と同じ画面にある | export コンポーネントが receipt コンポーネントと分離 |
| 状態の共有 | 選択状態（チェックボックス）を receipt 内で完結 | export が receipt の選択状態を参照するクロス依存 |

フロントエンドでは「ユーザーが見る画面」が自然な分割単位。バックエンドとの対応は API サービス関数で吸収する（`receiptService.ts` + `exportService.ts`）。

### 10.2 React Context の利用範囲

**方針:** サーバー状態には使わず、UI 横断の状態のみに限定する。

| 状態 | 管理方法 | 理由 |
|---|---|---|
| 通知（トースト） | Context | 画面をまたいで表示される UI 状態 |
| バッチ ID | URL パラメータ | ブックマーク可能、ブラウザバック対応 |
| フィルタ・ソート | Feature 内の useState | 画面ローカルな状態 |
| API レスポンスデータ | カスタムフック内の useState | Feature に閉じた状態 |

### 10.3 usePolling の共通化

バッチステータスのポーリング（画面B）とエクスポートステータスのポーリング（画面C）は、「一定間隔で API を呼び出し、条件を満たしたら停止する」という同一パターン。共通フックに抽出することで重複を避ける。

### 10.4 ページコンポーネントを薄く保つ

ページコンポーネント（`UploadPage` 等）は Feature コンポーネントの配置と URL パラメータの取得のみを担い、ビジネスロジックを持たない。ロジックはカスタムフックに集約する。
