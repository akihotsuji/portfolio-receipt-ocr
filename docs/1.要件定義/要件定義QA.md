# 領収書 OCR - 要件定義 Q&A

要件定義を進めるための問答記録です。  
回答が揃ったら、正式な要件定義書にまとめます。

---

## Q1. 機能要件（何を作るか）

### Q1-1. 領収書から何の情報を読み取りたいですか？

例: 日付、金額、店名、品目、税額、支払方法 など

**回答:**

以下の5項目を読み取る。確定申告（個人事業主）の帳簿記載に必要な情報がベース。

| # | 項目 | 優先度 | 備考 |
|---|---|---|---|
| 1 | 日付 | 必須 | 取引日 |
| 2 | 金額（合計） | 必須 | 支払総額 |
| 3 | 支払先（店名） | 必須 | 取引先名 |
| 4 | 品目（明細） | あれば | 何を買ったか |
| 5 | 税区分 | あれば | 8% / 10% など |

※ 精度は完璧でなくてよい。学習目的のため「それなりに読める」レベルで十分。

### Q1-2. 読み取り結果をどう使いたいですか？

例: 画面で確認・編集する、CSV でダウンロードする、JSON で出力する など

**回答:**

2つの独立したフローで利用する。

**フロー①: OCR 読み取り → 確認・編集 → 確定**
- アップロードした画像をプレビュー表示
- 同時に OCR 読み取り結果を編集可能なフォームで表示
- ユーザーが内容を確認・修正し「確定」するとサーバーに保存

**フロー②: データ一覧 → 選択 → エクスポート**
- 確定済みの読み取りデータが一覧で並ぶ
- 日時や品目でソート可能
- エクスポート対象を選んで出力（CSV 等）

### Q1-3. 1回の操作で何枚くらい処理する想定ですか？

例: 1枚ずつ、まとめて10枚程度、100枚以上 など

**回答:**

1回のアップロードで **最大10枚まで** 選択可能。

- 複数枚をまとめてアップロードし、サーバー側で非同期・並列に OCR 処理する
- 各画像ごとに処理状況（処理中 / 完了 / 失敗）を表示
- 並列処理の学習テーマにちょうどよい粒度

## Q2. 画面・操作フロー

### Q2-1. 基本的な操作フローのイメージはありますか？

例: アップロード → 処理中表示 → 結果一覧 → 編集 → エクスポート

**回答:**

3画面構成とする。

**画面A: アップロード画面**
- ファイルを選択（ドラッグ＆ドロップ or ボタン）
- 最大10枚まで同時選択可能
- アップロード開始 → 画面B へ遷移

**画面B: 処理状況 + 確認・編集画面**
- アップロードした画像の一覧と各ステータス（処理中 / 完了 / 失敗）を表示
- 完了した画像をクリック → 画像プレビュー＋読み取り結果の編集フォーム
- 内容を確認・修正して「確定」→ サーバーに保存

**画面C: データ一覧 + エクスポート画面**
- 確定済みの読み取りデータが一覧で並ぶ
- 日時・品目などでソート・フィルタ可能
- チェックボックスでエクスポート対象を選択 → CSV 出力

### Q2-2. 処理失敗時のユーザー体験はどうしたいですか？

例: 自動リトライ＋通知、手動で再試行ボタン、両方 など

**回答:**

**自動リトライ＋手動再試行** の二段構え。

1. OCR 処理が失敗した場合、バックエンド側で **自動リトライ**（最大3回）
2. 自動リトライですべて失敗した場合、画面B に「失敗」ステータスを表示
3. ユーザーが **手動で再試行ボタン** を押して再実行できる

学習テーマ「失敗系設計・リトライ・冪等性」を網羅する構成。

## Q3. 非機能要件（学習テーマ直結）

### Q3-1. ファイルアップロード

- 対応形式は？（JPEG, PNG, PDF など）
- 最大ファイルサイズは？
- 保存先は？（MinIO / ローカルストレージ）

**回答:**

- **対応形式**: JPEG, PNG, PDF
  - PDF: ダウンロードした電子領収書を想定
  - JPEG/PNG: 店舗レシートをスマホで撮影した写真を想定
- **最大ファイルサイズ**: 10MB / 枚
- **保存先**: MinIO（S3互換）に **専用バケット** を作成
  - 既存の `frontend-assets` バケットとは分離
  - `receipt-uploads` バケット（非公開）を新設
  - 領収書は個人データのため、署名付き URL で限定公開

### Q3-2. 非同期処理・並列処理

- OCR の同時処理数（N並列）の想定は？
- エクスポートの同時処理数（M並列）の想定は？

**回答:**

- **OCR 同時処理数: N=3**
  - 10枚アップロードされても同時に処理するのは3枚まで
  - 残りはキューで待機 → 完了次第順次処理
  - キューの滞留や処理遷移が目に見えて学習効果が高い
- **エクスポート同時処理数: M=3**
  - 同時に3つのエクスポートジョブまで実行可能
  - 個人利用で同時実行は稀だが、並列制御パターンの学習として実装
- 並列数の上限設定自体にはコストなし（リクエスト時のみリソース消費）

### Q3-3. 失敗系設計

- リトライ回数の上限は？
- タイムアウトの目安は？

**回答:**

- **自動リトライ: 最大3回**（Q2-2 で決定済み）
- **タイムアウト: 60秒**（画像・PDF 共通で統一）
- タイムアウト超過 → リトライ対象として扱う
- 3回リトライしても失敗 → 「失敗」ステータス＋手動再試行ボタン表示

## Q4. 外部依存

### Q4-1. OCR エンジンは何を使いたいですか？

選択肢:
- **Tesseract**（ローカル実行、無料、学習向き）
- **Google Cloud Vision API**（高精度、外部依存）
- **AWS Textract**（高精度、外部依存）
- その他

**回答:**

**Tesseract OCR** を採用する。

**採用理由:**
- 無料で利用可能（Apache 2.0 ライセンス）
- ローカル実行のため外部 API への依存なし
- Docker コンテナに含めることで、ローカル開発環境と AWS 本番環境で同一イメージが動く
- Tess4J（Java ラッパー）があり、Spring Boot と直接統合しやすい
- GPU 不要で軽量に動作
- 日本語の学習済みデータ（`jpn.traineddata`）が提供されており、追加学習は不要

**不採用の理由:**
- Google Cloud Vision / AWS Textract → 精度は高いが従量課金が発生し、学習目的にはオーバースペック
- NDLOCR-Lite → 日本語精度は高いが図書・雑誌向けに最適化されており、領収書用途とはユースケースが異なる。Java 連携も外部プロセス呼び出しとなり統合コストが高い

**補足:**
- PDF 対応のため、PDF → 画像変換には Apache PDFBox 等を併用する

## Q5. スコープ

### Q5-1. やらないことリスト（スコープ外）

学習目的に集中するため、スコープ外とするものを決めます。

例: 多言語対応、モバイル最適化、本番運用レベルのセキュリティ など

**回答:**

以下をスコープ外とする。

| スコープ外 | 理由 |
|---|---|
| 多言語対応 | 日本語の領収書のみ対象とする |
| モバイル専用UI | レスポンシブは最低限対応するが、ネイティブアプリは作らない |
| OCR 精度の追求 | Tesseract の標準精度で十分。チューニング・追加学習はしない |
| インボイス制度への完全対応 | 適格請求書番号の読み取り等は対象外。実務では重要だが、個人情報の取り扱いが重くなるため学習スコープから除外 |
| 会計ソフト連携 | CSV 出力までがスコープ。freee 等への API 連携はしない |
| 複数ページ PDF | 1ページ目のみ対象。複数ページ対応は処理が複雑になるため除外 |
