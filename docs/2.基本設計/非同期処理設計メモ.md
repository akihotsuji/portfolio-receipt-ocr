# 非同期処理フロー設計 - 議論メモ

## 設計の全体構成

```
【設計する論点】
1. キュー実装方式
2. ワーカー設計
3. リトライ戦略
4. タイムアウト設計
5. フロントエンドへのステータス通知方式
6. ローカル開発環境
```

---

## 1. キュー実装方式

**方針**: Amazon SQS を採用

| 項目 | 内容 |
|---|---|
| サービス | Amazon SQS（Standard Queue） |
| ローカル | LocalStack で代替 |
| OCR キュー | `receipt-ocr-queue` |
| エクスポートキュー | `receipt-export-queue` |

**決定事項**
- [x] 可視性タイムアウト（Visibility Timeout）: **120秒**
- [x] デッドレターキュー（DLQ）: **使う**（`receipt-ocr-queue-dlq`, `receipt-export-queue-dlq`）
- [x] DLQ への移動条件: `maxReceiveCount=3`（3回失敗で DLQ へ）

---

## 2. ワーカー設計

**方針**: Spring Boot アプリ内に SQS ポーリングワーカーを実装

```
[SQS Queue]
    ↓ @SqsListener（Spring Cloud AWS）でロングポーリング
[Worker スレッドプール]
    ↓ N並列で処理
[Tesseract OCR]
    ↓
[DB 更新 + MinIO/S3 保存]
```

**決定事項**
- [x] OCR 並列数: **N = 3**（要件定義より）
- [x] エクスポート並列数: **M = 3**（要件定義より）
- [x] ポーリング方式: **`@SqsListener`（Spring Cloud AWS）**

---

## 3. リトライ戦略

**前提**: 要件定義より OCR は最大3回リトライ

```
1回目失敗 → リトライ1回目
           → リトライ2回目
           → リトライ3回目
           → FAILED（手動リトライ API に切り替え）
```

**決定事項**
- [x] リトライのトリガー: **SQS の自動再配信に任せる**（例外スロー → メッセージ未削除 → 120秒後に再配信）

- [x] 手動リトライ: **上限なし**（ユーザーが何度でも再試行可能）

---

## 4. タイムアウト設計

**決定事項**
- [x] OCR タイムアウト: **60秒**（要件定義より）
- [x] タイムアウト時の挙動: **DB を FAILED に更新 → 例外再スロー → SQS 自動再配信に乗せる**（通常の失敗と同じ扱い）

- [x] エクスポートタイムアウト: **300秒（5分）**

---

## 5. フロントエンドへのステータス通知方式

フロントエンドは「OCR が完了したか」をどう知るか。

| 方式 | 概要 | メリット | デメリット |
|---|---|---|---|
| **ポーリング** | 定期的に GET /api/upload-batches/{batchId} を叩く | シンプル | 無駄なリクエストが発生 |
| **WebSocket** | 完了時にサーバーからプッシュ | リアルタイム | 実装が複雑 |
| **SSE** | Server-Sent Events でプッシュ | WebSocket より実装が軽い | 一方向通信のみ |

**決定事項**
- [x] 通知方式: **フロントエンドポーリング**（定期的に `GET /api/upload-batches/{batchId}` を叩く）

- [x] ポーリング間隔: **2秒**

---

## 6. ローカル開発環境

**方針**: LocalStack を Docker で起動して SQS をエミュレート

**決定事項**
- [x] LocalStack の導入範囲: **SQS のみ**（S3 は MinIO で代替済み）

- [x] docker-compose: **LocalStack コンテナを追加**（詳細は実装時に決定）
